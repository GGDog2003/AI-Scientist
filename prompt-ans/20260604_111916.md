# 本次答复

整体上更偏 `Plan-and-Solve`，不是纯 `ReAct`。

更准确地说，是：

- 系统级编排：`Plan-and-Solve`
- 节点内工具调用：可以局部使用 `ReAct`

原因是你当前这份科研 Agent 方案有几个明显特征：

- 先定义阶段目标，再按状态机推进
- 有明确的阶段拆分：文献调研、创新点评审、实验设计、论文写作、盲审返修
- 每一步都有结构化产物和状态转移条件
- 强依赖 `LangGraph StateGraph` 做流程控制

这些都更符合 `Plan-and-Solve`，而不是经典 `ReAct` 的“边想边调工具边前进”单 Agent 范式。

如果进一步细分：

- `StudentAgent` 在“读论文 -> 抽取信息 -> 写笔记”这种局部任务里，可以用 `ReAct`
- 整个多 Agent 科研系统的总工作流，应该用 `Plan-and-Solve + 状态机`

所以最合适的说法是：

`整体是 Plan-and-Solve，局部节点可以嵌 ReAct。`
