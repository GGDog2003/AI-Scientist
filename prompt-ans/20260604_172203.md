这次已经不是密钥问题了，而是模型兼容性问题。`deepseek-v4-flash` 当前这条链路下不支持你现在这类 `tool_choice` 组合，所以原先用 `create_agent(..., tools=..., response_format=...)` 的调用方式会直接被拒绝，报出 `Thinking mode does not support this tool_choice`。

我已经重写了 [app/agents/base_agent.py](/D:/python/AI-Scientist/app/agents/base_agent.py)。现在逻辑变成两层：

1. 先优先尝试原来的 `Agent + Tools + structured response` 路径。  
2. 如果命中 `tool_choice` 兼容性错误，就自动回退到“无工具的直接结构化输出模式”，也就是 `model.with_structured_output(...)`，这样先保证文献汇总、创新点、实验方案、论文草稿这些结构化阶段能继续跑下去。

这相当于保留了兼容模型上的 ReAct 能力，同时对 DeepSeek 这类不兼容当前工具选择语义的模型自动降级处理，不再一上来就直接炸死在第一步。

改了什么：把 Agent 层改成“工具模式优先，失败后自动回退到无工具结构化输出模式”，专门兼容 DeepSeek 的 `tool_choice` 限制。  
有没有验证：`python -m compileall main.py app tests` 已通过。  
还剩什么风险：这次回退后，`summarize_literature` 等阶段虽然能继续用真实模型跑，但不会在这些阶段里真正调用工具读取 PDF 文本；也就是说，它先解决的是“流程跑不起来”的问题，不是“文献解析质量最佳化”的问题。下一步如果你要，我应该继续把文献阅读阶段改成显式先解析 PDF，再把解析结果喂给模型，而不是依赖 Agent 现场调工具。
