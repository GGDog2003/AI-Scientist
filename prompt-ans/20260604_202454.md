这次已经定位到第三层兼容性问题了：DeepSeek 这条链路不只是不支持 `tool_choice`，连 `response_format` 也不支持。所以之前即使从 `create_agent(...response_format=...)` 回退到了 `with_structured_output(...)`，本质上还是会继续命中同类限制。

我已经把 [app/agents/base_agent.py](/D:/python/AI-Scientist/app/agents/base_agent.py) 再降一级重写了。现在结构化调用顺序变成：

1. 先尝试原来的 `Agent + Tools + response_format` 路径。  
2. 如果命中 `tool_choice` 或 `response_format` 兼容性错误，就回退到“普通聊天调用”。  
3. 在普通聊天调用里，系统会强制要求模型“只能输出合法 JSON”，然后由本地 `json.loads + Pydantic` 做解析和校验。  

这样就彻底绕开了 DeepSeek 当前不支持的 `tool_choice` 和 `response_format` 两套机制，至少结构化阶段可以继续走下去。

改了什么：把 Agent 层的最终兜底改成“纯文本 JSON 输出 + 本地解析”，不再依赖 `with_structured_output`。
