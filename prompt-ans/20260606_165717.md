这次我把问题拆成了两条线处理：一条是“偶发 JSON 解析失败”，另一条是“开始撰写后界面明显卡顿”。根因最终不是同一个点，但都出在 GUI 状态同步链路上。

改了什么

- 重写了 [ui/src-tauri/src/lib.rs](/D:/python/AI-Scientist/ui/src-tauri/src/lib.rs) 的 `inspect_workflow`：
  - 以前每 1.5 秒轮询一次，都会重新起一个 Python 进程，再重建整套工作流对象去查状态；
  - 现在改成 Rust 直接读取 `workspace/logs/progress.jsonl` 和 `workspace/logs/workflow.jsonl`，本地拼装 `WorkflowSnapshot` 返回给前端。
- 给日志读取加了并发容错：
  - 如果 `progress.jsonl` 正好处于写入中，读到半截 JSON 行时会自动跳过坏行，不再直接抛解析异常。
- 保留了 `start_workflow` / `resume_workflow` 走 Python 的方式，但把高频轮询链路彻底从 Python 进程调用中拆掉了。

有没有验证

- `cargo check` 已在 [ui/src-tauri](/D:/python/AI-Scientist/ui/src-tauri) 通过。
- `npm run build` 已在 [ui](/D:/python/AI-Scientist/ui) 通过。
- 本地检查了 `workspace/logs/progress.jsonl` 当前内容，确认真实文件本身是合法 JSONL，只是原先高频轮询链路更容易在并发时读到不完整内容。

还剩什么风险

- 现在 `inspect_workflow` 为了减轻卡顿，主要依赖日志推断状态，没有再实时读取完整 LangGraph 快照，所以 `artifacts/messages_log/final_summary` 这些字段会比之前简化。
- 如果你后面希望 GUI 里也完整展示所有通信记录，最好下一步在 Python 工作流每次节点结束后额外落一个稳定的 `state_snapshot.json`，然后 GUI 只读这个文件。
- 你必须重启当前正在运行的 `Tauri` 进程，这次的关键改动都在 Rust 后端，旧窗口不会热更新。
